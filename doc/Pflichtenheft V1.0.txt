Pflichtenheft – Template Service
Pflichtenheft V1.0
Service-Name: 	template-service
Version: 		1.0
Stand: 		15.02.2026
Geltungsbereich: 	Microservice zur Verwaltung, Versionierung, Freigabe und zum 				Rendering von Templates (Text/HTML) innerhalb eines Multi-Tenant 			SaaS-Portals inkl. Plattform-Templates (GLOBAL), Tenant-Templates 			(TENANT), Placeholder-Extraktion, Validierung, Preview sowie 				stabilem Error-Modell und OpenAPI/Swagger-Dokumentation.
1. Einleitung
1.1 Ziel
Ziel ist ein zentraler Template-Service, der Templates persistiert, versioniert, freigibt (Approval) und rendert (Text/HTML) – mandantenfähig und auditierbar. Der Service stellt eine konsistente REST-API bereit (unter /api/v1/template/*) und liefert gerenderte Inhalte für andere Services (z. B. Messaging / Document-Service).
Der Service stellt insbesondere sicher:
Verwaltung von Tenant-Templates je Mandant (voll editierbar & versionierbar)
Verwaltung von Plattform-Templates (GLOBAL) durch Portal-Besitzer (read-only für Tenants)
Tenant Version Pinning für GLOBAL Templates (Tenant wählt freigegebene Version)
Versionierung (immutable Versions) mit Statusmodell (DRAFT/APPROVED/DEPRECATED/ARCHIVED)
Rendering in TEXT und HTML (HTML: escaped by default)
Placeholder-Extraktion ({{key}}) + MissingKeyPolicy
Preview/Validate für UI/Portal-Administrationsoberfläche
OpenAPI/Swagger DoD: kein Endpoint gilt als fertig ohne vollständige Swagger-Doku

1.2 Abgrenzung / Nicht-Ziele (MVP)
Nicht Bestandteil des MVP:
Authentifizierung/Token-Ausstellung ? eigener Auth-Service (Resource Server validiert JWT)
Zustellung/Versand von Nachrichten ? Messaging/Channel-Services
Erzeugung von PDFs/Dokumenten ? eigener Document-Service (später)
Quota/Billing ? separater Service (optional später)
Vollständiges WYSIWYG-Editor-Frontend ? Portal/Frontend (nicht Template-Service)
“Template overrides” der GLOBAL Templates durch Tenants ? nicht erlaubt (No Shadowing)
1.3 Begriffe
TemplateFamily: Template-“Familie” (Identität über Key+Channel+Locale+Scope/Owner)
TemplateVersion: konkrete Version einer Familie (immutable)
Scope: GLOBAL (Portal-Besitzer) oder TENANT (Mandant)
Pinned Version: vom Tenant ausgewählte freigegebene GLOBAL Template-Version (in Portal-Konfig)
RenderTarget: TEXT oder HTML (optionale Erweiterung: MARKDOWN)
MissingKeyPolicy: FAIL | KEEP_TOKEN | EMPTY
Placeholders: extrahierte Keys aus {{key}}

3. Fachliche Anforderungen
3.1 Mandantenmodell / Tenant-Kontext (MUSS)
Der Tenant-Kontext kommt aus dem JWT Claim tenant_id.
Der Service akzeptiert keinen tenantId aus Body/Query als Quelle der Wahrheit.
Jeder Datensatz ist scope-/owner-scharf (siehe 3.2).
3.2 Scope/Ownership Modell (MUSS)
scope = GLOBAL ? ownerTenantId = "__GLOBAL__" (reserviert)
scope = TENANT ? ownerTenantId = <tenant_id aus JWT>
Tenants dürfen GLOBAL Templates nutzen, aber nicht bearbeiten/überschreiben/versionieren.
3.3 IDs & globale Eindeutigkeit (MUSS)
templateId, versionId sind UUID/ULID Strings.
Zugriff MUSS prüfen: Objekt gehört zum ownerTenantId und scope gemäß Berechtigung.
3.4 TemplateFamily Felder (MUSS/SOLL)
Entity: TemplateFamily
MUSS:
templateId
scope (GLOBAL/TENANT)
ownerTenantId ("__GLOBAL__" oder Tenant)
templateKey (Alias/Key)
channel (EMAIL/SMS/WHATSAPP/IN_APP/…)
locale (z. B. de-DE)
category (TRANSACTIONAL/ALERT/MARKETING/SYSTEM)
activeApprovedVersion (int oder versionRef)
Audit: createdAt/By, modifiedAt/By
version (Optimistic Locking)
SOLL:
description, tags[]
changeLog (Kurzbeschreibung pro Version optional)
3.5 TemplateVersion Felder (MUSS/SOLL)
Entity: TemplateVersion
MUSS:
versionId
templateId
versionNo (1..n, monoton steigend)
status (DRAFT/APPROVED/DEPRECATED/ARCHIVED)
renderTarget (TEXT/HTML)
subjectTpl (optional; MUSS bei EMAIL)
bodyTpl (MUSS)
placeholders[] (extrahiert aus {{key}})
missingKeyPolicyDefault (optional; Standard je Environment)
Audit: createdAt/By
version (Optimistic Locking)
SOLL:
requiredKeys[] (wenn nicht alle placeholders required sind)
defaults (Map key?default)
compatibilityLevel (MAJOR/MINOR/PATCH oder breakingChange flag)
sunsetAt (für Deprecation/Ende der Nutzbarkeit)
3.6 Placeholder-Regeln (MUSS)
Platzhalter-Syntax: {{key}} (Whitelist für key z. B. [a-zA-Z0-9_.-]+)
Beim Speichern/Ändern einer Version:
Placeholders werden extrahiert und gespeichert
Extra Variables sind erlaubt (werden ignoriert)
3.7 Rendering (MUSS)
Render API erzeugt Ausgabe aus:
Template (templateId oder templateKey+channel+locale+scope)
Version (explizit; oder „activeApprovedVersion“)
variables: Map<String,Object>
MissingKeyPolicy:
FAIL ? Render bricht ab, liefert MISSING_KEYS
KEEP_TOKEN ? Token bleibt im Output + warning
EMPTY ? Token wird leer ersetzt + warning
3.8 Escaping / Output-Sicherheit (MUSS)
renderTarget=HTML: escaped by default
renderTarget=TEXT: Normalisierung (Zeilenumbrüche, Steuerzeichen entfernen/konform)
Keine “raw/unescaped” Ausgabe im MVP (optional später als bewusstes Feature)
3.9 Locale-/Fallback Regeln (MUSS/SOLL)
MUSS:
Exakter Locale-Match (de-DE)
Fallback: Sprache (de) ? DefaultLocale
SOLL:
Tenant/Portal DefaultLocale konfigurierbar

3.10 Versionierung & Lifecycle (MUSS)
Versions sind immutable (kein In-Place Edit)
Approval:
Nur APPROVED ist produktiv renderbar
Approve setzt activeApprovedVersion (pro TemplateFamily genau eine aktive APPROVED)
Deprecation:
DEPRECATED bleibt renderbar (bis sunsetAt, falls gesetzt)
Archived:
ARCHIVED ist nicht renderbar
3.11 GLOBAL Templates & Tenant Pinning (MUSS)
Plattform erstellt GLOBAL Templates + Versionen
Tenants dürfen:
verfügbare APPROVED Versionen ansehen
im Portal eine APPROVED Version pinnen (Portal-Konfig, nicht Template-Service)
Tenants dürfen nicht:
GLOBAL Template bearbeiten
GLOBAL Versionen erstellen/approve/deprecate
GLOBAL Template durch eigenes Template überschreiben (No Shadowing)
3.12 No Shadowing / Key Reservation (MUSS)
Keys im Namespace platform.*/system.* sind reserviert
Tenant darf keine TemplateFamily anlegen, die ein GLOBAL Template “überschreibt”
Regel: identische Kombination (templateKey, channel, locale) ist im TENANT Scope verboten, wenn im GLOBAL Scope vorhanden
3.13 Preview & Validate (MUSS)
Preview: rendert auch DRAFT Versionen (für Admin UI)
Validate: prüft Syntax, extrahiert placeholders, meldet required/missing, channel constraints

3.14 Sicherheit / Zugriff (MUSS)
JWT Resource Server validiert Token
Rollen (mindestens):
platform_admin: GLOBAL Verwaltung
tenant_admin: TENANT Verwaltung im eigenen tenant
user: Render/Preview (je nach Policy)
Keine sensiblen Daten in Logs (variables maskieren/keine Vollpayload-Logs)
3.15 Domain Events / Outbox (SOLL, empfohlen)
Optional: Outbox Pattern für Events:
TemplateCreated/Updated
TemplateVersionCreated/Approved/Deprecated/Archived
RenderRequested (optional, ohne variables im Klartext)

6. REST-API Spezifikation – Basis /api/v1/template
6.1 Rendering / Preview / Validation
POST /api/v1/template/render
Zweck: produktives Rendern (nur APPROVED)
POST /api/v1/template/preview
Zweck: Rendern für UI (DRAFT erlaubt)
POST /api/v1/template/validate
Zweck: Syntax/Placeholder/Constraints prüfen

6.2 TemplateFamily Verwaltung (TENANT)
POST /api/v1/template/templates
Zweck: TENANT TemplateFamily anlegen
GET /api/v1/template/templates/{templateId}
Zweck: TemplateFamily laden
GET /api/v1/template/templates?scope=TENANT&...
Zweck: listen (Paging/Filter)
PUT /api/v1/template/templates/{templateId}
Zweck: Metadaten ändern (nicht Inhalt)
6.3 TemplateFamily Verwaltung (GLOBAL)
Identische Endpoints, aber nur für platform_admin und scope=GLOBAL.
6.4 Versionen
POST /api/v1/template/templates/{templateId}/versions
Zweck: neue DRAFT Version erstellen
GET /api/v1/template/templates/{templateId}/versions/{versionNo}
Zweck: Version laden
GET /api/v1/template/templates/{templateId}/versions?status=...
Zweck: Versionen listen
POST /api/v1/template/templates/{templateId}/versions/{versionNo}/approve
Zweck: Version freigeben (setzt activeApprovedVersion)
POST /api/v1/template/templates/{templateId}/versions/{versionNo}/deprecate (SOLL)
POST /api/v1/template/templates/{templateId}/versions/{versionNo}/archive (SOLL)
6.5 Portal-Unterstützung (SOLL)
GET /api/v1/template/catalog?scope=GLOBAL&channel=...&locale=...
Zweck: Portal kann verfügbare GLOBAL Templates anzeigen
GET /api/v1/template/catalog/{templateId}/approved-versions
Zweck: Portal kann verfügbare APPROVED Versionen abrufen
Pinning selbst liegt im Portal (Binding), nicht im Template-Service.

11. Abnahmekriterien (Auszug)
TemplateFamily (TENANT/GLOBAL) kann angelegt, geändert, gelistet, geladen werden (Audit korrekt)
Versionen können erstellt und APPROVED gesetzt werden
Rendern ist nur für APPROVED produktiv möglich; Preview erlaubt DRAFT
Placeholders werden extrahiert und MissingKeyPolicy korrekt umgesetzt
No Shadowing/Key Reservation wird erzwungen
Mandantentrennung wird erzwungen (403 bei Cross-Tenant)
Swagger/OpenAPI ist vollständig (DoD)
Actuator Health erreichbar
Dockerfile + Compose funktionieren lokal
