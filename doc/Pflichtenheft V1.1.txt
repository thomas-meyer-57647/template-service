Pflichtenheft ñ Template Service (V1.1)
Service-Name: template-service
Version: 1.1
Stand: 17.02.2026
Geltungsbereich: Microservice zur Verwaltung, Versionierung, Freigabe (Approval) und zum Rendering von Templates (Text/HTML) innerhalb eines Multi-Tenant SaaS-Portals inkl. Plattform-Templates (GLOBAL) und Tenant-Templates (TENANT). Der Service integriert sich in das IAM/Auth-Konzept via JWT (Pattern A: Rechte/Scopes im Token) und erzwingt Mandantentrennung ueber tenant_id.

1. 0. Aenderungshistorie (Auszug)
V1.1 (17.02.2026)
- Security: Umstellung auf JWT (Bearer Token) gem. IAM/Auth-Konvention (Resource Server). Rechte/Scopes fuer den Tenant sind im JWT enthalten (Pattern A).
- Audience: aud muss template-service enthalten (strict).
- Tenant-Isolation: tenant_id aus JWT ist Quelle der Wahrheit (Model 1). TenantId aus Body/Query ist nicht zulaessig als Steuerungsfeld.
- Actor/Request-Kontext: actor wird aus JWT (sub) abgeleitet; RequestId wird in Logs/Fehlern genutzt.
- Locale-Aufloesung: Sprache/Locale wird als REST-Attribut im Render/Preview/Resolve genutzt; Fallback-Regeln (exakt -> Sprache -> Default) sind verbindlich.
- Scopes/Permissions: template:* steuern Zugriff auf Admin-/Read-/Runtime-Endpunkte.
2. 1. Einleitung
1.1 Ziel
Ziel ist ein zentraler Template-Service, der Templates persistiert, versioniert, freigibt (Approval) und rendert (Text/HTML) ñ mandantenfaehig und auditierbar. Der Service stellt eine konsistente REST-API bereit (unter /api/v1/template/*) und liefert gerenderte Inhalte fuer andere Services (z. B. Messaging/Notification-Service).
Der Service stellt insbesondere sicher:
Verwaltung von Tenant-Templates je Mandant (voll editierbar & versionierbar)
Verwaltung von Plattform-Templates (GLOBAL) durch Portal-Besitzer (read-only fuer Tenants)
Tenant Version Pinning fuer GLOBAL Templates (Tenant waehlt freigegebene Version im Portal; nicht im Template-Service)
Versionierung (immutable Versions) mit Statusmodell (DRAFT/APPROVED/DEPRECATED/ARCHIVED)
Rendering in TEXT und HTML (HTML: escaped by default)
Placeholder-Extraktion ({{key}}) + MissingKeyPolicy
Preview/Validate fuer UI/Portal-Administrationsoberflaeche
Security via JWT/IAM (Bearer Token, Scopes, Tenant-Isolation; Pattern A)
1.2 Abgrenzung / Nicht-Ziele (MVP)
Nicht Bestandteil des MVP:
Authentifizierung/Token-Ausstellung (Auth-Service stellt Tokens aus; template-service ist Resource Server)
Zustellung/Versand von Nachrichten (Messaging/Channel-Services)
Erzeugung von PDFs/Dokumenten (Document-Service spaeter)
Quota/Billing (separater Service spaeter)
Vollstaendiges WYSIWYG-Editor-Frontend (Portal/Frontend)
Template overrides der GLOBAL Templates durch Tenants (nicht erlaubt; No Shadowing)
1.3 Begriffe (Mini-Glossar)
TemplateFamily: Template-"Familie" (Identitaet ueber Key+Channel+Locale+Scope/Owner)
TemplateVersion: konkrete Version einer Familie (immutable = nach dem Speichern nicht mehr in-place editierbar)
Scope: GLOBAL (Portal-Besitzer) oder TENANT (Mandant)
Pinned Version: vom Tenant ausgewaehlte freigegebene GLOBAL Template-Version (in Portal-Konfig, nicht Template-Service)
RenderTarget: TEXT oder HTML (optional spaeter: MARKDOWN)
MissingKeyPolicy: FAIL | KEEP_TOKEN | EMPTY
Locale: Sprach-/Regionscode (BCP-47-aehnlich), z. B. de, de-DE, en-GB
JWT / Access Token: signiertes Bearer Token vom Auth-Service; enthaelt sub (User), tenant_id (Mandant), scope/scp (Berechtigungen), exp (Ablauf)

4. 3. Security-Integration (JWT/IAM) (MUSS)
3.1 Grundprinzip (Pattern A)
Der template-service ist ein Resource Server und validiert JWT Access Tokens gemaess IAM/Auth-Konvention. Die benoetigten Berechtigungen (Scopes/Permissions) fuer den Tenant sind bereits im JWT enthalten (Pattern A). Der Service macht keine Token-Introspection pro Request und stellt keine Tokens aus.
3.2 Pflicht-Claims (MUSS)
Der Service erwartet mindestens folgende Claims im JWT:
iss (Issuer)
sub (Subject/User ID)
aud (Audience) ñ muss template-service enthalten (strict)
exp (Ablaufzeitpunkt), iat (Ausstellungszeitpunkt)
jti (Token ID)
tenant_id (Mandant; Model 1)
3.3 Audience (MUSS)
aud muss template-service enthalten (strict). Tokens ohne passende Audience werden abgewiesen (401).
3.4 Scopes/Permissions (MUSS)
Scopes werden ueber den Claim scope (String, space-separated) oder scp (Array) transportiert.
Minimal erforderliche Scopes:
template:read ñ lesen/listen/resolve + render (produktiver Zugriff)
template:admin ñ TENANT Verwaltung (Families/Versions/Approve) im eigenen tenant
template:global:admin ñ GLOBAL Verwaltung (nur Plattform-Besitzer)
Hinweis: Rollen (z. B. tenant_admin, platform_admin) koennen existieren, jedoch ist die Autorisierungsquelle im Service die Scope-Pruefung.
3.5 Tenant-Isolation (MUSS, Model 1)
Mandantenkontext wird ausschliesslich aus tenant_id im Token abgeleitet (Quelle der Wahrheit).
Der Service akzeptiert keinen tenantId aus Body/Query als Steuerungsfeld.
Alle Datenzugriffe (TENANT) sind ownerTenantId-scharf auf tenant_id.
3.6 Konfiguration (SOLL)
Konfiguration (Beispiele):
security.jwt.issuer-uri oder security.jwt.jwks-uri
security.jwt.audience = template-service (strict)
security.jwt.clock-skew-seconds (kleiner Toleranzbereich)
Legacy/KANN: Ein Admin-API-Key kann fuer Dev/Legacy optional konfiguriert werden, Standard ist JWT.
5. 4. Fachliche Anforderungen
4.1 Mandantenmodell / Tenant-Kontext (MUSS)
Tenant-Kontext kommt aus JWT Claim tenant_id.
Kein tenantId aus Body/Query als Quelle der Wahrheit.
scope-/owner-scharfe Speicherung (siehe 4.2).
4.2 Scope/Ownership Modell (MUSS)
scope = GLOBAL -> ownerTenantId = "__GLOBAL__" (reserviert)
scope = TENANT -> ownerTenantId = <tenant_id aus JWT>
Tenants duerfen GLOBAL Templates nutzen, aber nicht bearbeiten/ueberschreiben/versionieren.
4.3 TemplateFamily (MUSS/SOLL)
Entity: TemplateFamily
MUSS-Felder:
templateId
scope (GLOBAL/TENANT)
ownerTenantId ("__GLOBAL__" oder tenant_id)
templateKey
channel (EMAIL/SMS/WHATSAPP/IN_APP/...)
locale (z. B. de-DE)
category (TRANSACTIONAL/ALERT/MARKETING/SYSTEM)
activeApprovedVersion (int oder Referenz)
Audit: createdAt/By, modifiedAt/By
version (Optimistic Locking)
SOLL-Felder:
description
tags[]
changeLog (Kurzbeschreibung pro Version optional)
4.4 TemplateVersion (MUSS/SOLL)
Entity: TemplateVersion
MUSS-Felder:
versionId
templateId
versionNo (1..n, monoton steigend)
status (DRAFT/APPROVED/DEPRECATED/ARCHIVED)
renderTarget (TEXT/HTML)
subjectTpl (optional; MUSS bei EMAIL)
bodyTpl (MUSS)
placeholders[] (extrahiert aus {{key}})
missingKeyPolicyDefault (optional)
Audit: createdAt/By
version (Optimistic Locking)
SOLL-Felder:
requiredKeys[] (wenn nicht alle placeholders required sind)
defaults (Map key->default)
compatibilityLevel (MAJOR/MINOR/PATCH oder breakingChange flag)
sunsetAt (Ende der Nutzbarkeit)
4.5 Placeholder-Regeln (MUSS)
Platzhalter-Syntax: {{key}} (Whitelist z. B. [a-zA-Z0-9_.-]+)
Beim Speichern/Anlegen einer Version werden Placeholders extrahiert und gespeichert.
Extra Variables sind erlaubt (werden ignoriert).
4.6 Rendering (MUSS)
Render API erzeugt Ausgabe aus:
Template-Identitaet: entweder templateId ODER (templateKey + channel + locale + scope)
Version: explizit oder automatisch activeApprovedVersion (produktiver Standard)
variables: Map<String,Object>
missingKeyPolicy: FAIL | KEEP_TOKEN | EMPTY
MissingKeyPolicy Semantik:
FAIL: Render bricht ab und liefert Fehlercode MISSING_KEYS.
KEEP_TOKEN: Token bleibt im Output und wird als Warning gemeldet.
EMPTY: Token wird leer ersetzt und wird als Warning gemeldet.
4.7 Escaping / Output-Sicherheit (MUSS)
renderTarget=HTML: escaped by default (Schutz vor HTML/Script-Injection).
renderTarget=TEXT: Normalisierung (Zeilenumbrueche, Steuerzeichen entfernen/konform).
Keine raw/unescaped Ausgabe im MVP.
4.8 Locale-/Fallback Regeln (MUSS/SOLL)
MUSS:
Exakter Locale-Match (z. B. de-DE).
Fallback: Sprache (de) -> DefaultLocale.
SOLL:
Tenant/Portal DefaultLocale konfigurierbar (z. B. via i18n-Service oder Portal-Setting).
Optional Regional-Fallback (Backlog): de-CH -> de-DE -> en-GB.
4.9 Versionierung & Lifecycle (MUSS)
Versions sind immutable (kein In-Place Edit). Aenderungen erfolgen durch neue Version.
Nur APPROVED ist produktiv renderbar.
Approve setzt activeApprovedVersion (pro TemplateFamily genau eine aktive APPROVED).
DEPRECATED bleibt renderbar (bis sunsetAt, falls gesetzt).
ARCHIVED ist nicht renderbar.
4.10 GLOBAL Templates & Tenant Pinning (MUSS)
Plattform erstellt GLOBAL Templates + Versionen.
Tenants duerfen APPROVED Versionen ansehen und im Portal eine APPROVED Version pinnen.
Pinning selbst liegt im Portal (Binding), nicht im Template-Service.
Tenants duerfen GLOBAL Templates nicht bearbeiten oder Versionen erstellen/approve/deprecate.
4.11 No Shadowing / Key Reservation (MUSS)
Keys im Namespace platform.*/system.* sind reserviert.
Tenant darf keine TemplateFamily anlegen, die ein GLOBAL Template ueberschreibt.
Regel: identische Kombination (templateKey, channel, locale) ist im TENANT Scope verboten, wenn im GLOBAL Scope vorhanden.
4.12 Preview & Validate (MUSS)
Preview: rendert auch DRAFT Versionen (fuer Admin UI).
Validate: prueft Syntax, extrahiert placeholders, meldet required/missing, channel constraints.
locale wird als Request-Attribut uebergeben und bei Resolve/Fallback beruecksichtigt.
4.13 Domain Events / Outbox (SOLL, empfohlen)
Optional (SOLL): Outbox Pattern fuer Events:
TemplateCreated/Updated
TemplateVersionCreated/Approved/Deprecated/Archived
RenderRequested (optional, ohne variables im Klartext)

7. 6. REST-API Spezifikation (Basis: /api/v1/template)
6.1 Security-Konvention (MUSS)
Jeder Request traegt (Standard): Authorization: Bearer <access_token>. Scopes werden je Endpoint erzwungen.
6.2 Rendering / Preview / Validation
POST /api/v1/template/render ñ produktives Rendern (nur APPROVED) (Scope: template:read)
POST /api/v1/template/preview ñ Rendern fuer UI (DRAFT erlaubt) (Scope: template:read oder template:admin)
POST /api/v1/template/validate ñ Syntax/Placeholder/Constraints pruefen (Scope: template:admin)
Hinweis: Sprache/Locale wird im Request als Attribut uebergeben (locale) und fuer Resolve/Fallback verwendet.
6.3 TemplateFamily Verwaltung (TENANT)
POST /api/v1/template/templates ñ TENANT TemplateFamily anlegen (Scope: template:admin)
GET /api/v1/template/templates/{templateId} ñ TemplateFamily laden (Scope: template:read)
GET /api/v1/template/templates?scope=TENANT&... ñ listen (Paging/Filter) (Scope: template:read)
PUT /api/v1/template/templates/{templateId} ñ Metadaten aendern (nicht Inhalt) (Scope: template:admin)
6.4 TemplateFamily Verwaltung (GLOBAL)
Identische Endpoints wie 6.3, aber scope=GLOBAL und Scope: template:global:admin
6.5 Versionen
POST /api/v1/template/templates/{templateId}/versions ñ neue DRAFT Version erstellen (Scope: template:admin bzw. template:global:admin)
GET /api/v1/template/templates/{templateId}/versions/{versionNo} ñ Version laden (Scope: template:read)
GET /api/v1/template/templates/{templateId}/versions?status=... ñ Versionen listen (Paging) (Scope: template:read)
POST /api/v1/template/templates/{templateId}/versions/{versionNo}/approve ñ Version freigeben (Scope: template:admin bzw. template:global:admin)
POST /api/v1/template/templates/{templateId}/versions/{versionNo}/deprecate ñ Version deprecaten (SOLL)
POST /api/v1/template/templates/{templateId}/versions/{versionNo}/archive ñ Version archivieren (SOLL)
6.6 Catalog (SOLL)
GET /api/v1/template/catalog?scope=GLOBAL&channel=...&locale=... ñ Portal zeigt verfuegbare GLOBAL Templates
GET /api/v1/template/catalog/{templateId}/approved-versions ñ verfuegbare APPROVED Versionen

12. 11. Abnahmekriterien (Auszug)
TemplateFamily (TENANT/GLOBAL) kann angelegt, geaendert, gelistet, geladen werden (Audit korrekt).
Versionen koennen erstellt und APPROVED gesetzt werden (immutable).
Rendern ist nur fuer APPROVED produktiv moeglich; Preview erlaubt DRAFT.
Placeholders werden extrahiert und MissingKeyPolicy korrekt umgesetzt.
No Shadowing/Key Reservation wird erzwungen.
Mandantentrennung wird erzwungen (403 bei Cross-Tenant).
JWT-Validierung (Signature, iss, aud, exp) funktioniert; fehlende/ungueltige Tokens -> 401.
Scopes werden erzwungen: template:read vs template:admin vs template:global:admin.
Swagger/OpenAPI ist vollstaendig (DoD).
Actuator Health erreichbar.
Dockerfile + Compose funktionieren lokal.
